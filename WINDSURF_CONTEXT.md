# WINDSURF SESSION CONTEXT

## PROJECT OVERVIEW
**Project Name:** ConstructionOpsStanton  
**Purpose:** Construction project management system for tracking projects, budgets, contractors, payments, and daily operations  
**Users:** Construction company staff managing multi-family renovation projects, tracking budgets, contractor payments, and project progress  
**Scale:** Multi-project portfolio management with entity-based ownership structure

---

## TECH STACK
```
Frontend:      Next.js 14 (App Router)
Styling:       TailwindCSS with CSS variables for theming
State:         React Context (DataContext, ProjectContext, ModalContext)
Backend:       Next.js API Routes
Database:      PostgreSQL via Supabase
Auth:          Supabase Auth with role-based access
Key Libraries: lucide-react (icons), shadcn/ui components, frappe-gantt (scheduling)
```

## DATA MODEL

### Core Entities
```
projects - Main construction projects
  - id, name, client_name, address, budget, status
  - owner_entity_id â†’ owner_entities
  - current_phase, start_date, target_completion_date

owner_entities - LLC/ownership entities
  - id, name, entity_type, tax_id, contact_info
  - Properties owned through projects

contractors - Subcontractors/vendors
  - id, name, trade, phone, email, status

property_budgets - Budget line items per project
  - id, project_id â†’ projects
  - category_name, budgeted_amount, actual_amount

contracts - Contractor assignments to projects
  - id, project_id â†’ projects
  - contractor_id â†’ contractors
  - contract_amount, status

payment_applications - Payment requests from contractors
  - id, project_id, contractor_id
  - amount_requested, status, pm_notes

change_orders - Budget change requests
  - id, project_id, contractor_id
  - cost_impact, status, approval workflow

punch_list_items - QC/deficiency tracking
  - id, project_id, description, severity, status
  - assigned_to â†’ contractors

daily_log_requests - Automated PM note collection
  - id, project_id, pm_phone_number, request_time

loans - Construction loan tracking
  - id, project_id, lender, loan_amount
  - draws, budget item allocations

schedule_tasks - Gantt chart tasks
  - id, project_id, name, start_date, end_date
  - dependencies, budget_item_id
```

### Key Relationships
- **Projects** belong to **Owner Entities** (1:many)
- **Contractors** work on multiple **Projects** via **Contracts** (many:many)
- **Payment Applications** link **Contractors** to **Projects** with line items
- **Change Orders** can affect **Budget Categories** and **Contractors**
- **Schedule Tasks** can link to **Budget Items** for cost tracking

## ARCHITECTURE PATTERNS

### Data Fetching
- Server components fetch via Supabase client
- Client components use API routes (`/api/*`) with auth headers
- DataContext provides cached contractors, projects, subcontractors globally

### State Management
- **Server State:** Fetched via API routes, cached in component state
- **UI State:** Local useState for modals, forms, filters
- **Global State:** React Context for shared data (projects, contractors, user role)

### API/Backend Logic
- Business logic in API routes (`src/app/api/`)
- Database operations via Supabase client
- Auth validation via `supabase.auth.getSession()`

## DOMAIN TERMINOLOGY

| Term | Meaning |
|------|---------|
| **Owner Entity** | LLC or legal entity that owns properties/projects |
| **Change Order (CO)** | Request to modify project scope/budget |
| **Payment Application** | Contractor's request for payment with line items |
| **Punch List** | QC deficiency items to be fixed before project completion |
| **Draw** | Disbursement from construction loan |
| **PM Notes** | Daily progress notes from Project Manager |
| **Budget Category** | Line item in project budget (e.g., "Electrical", "Plumbing") |
| **Verification** | PM review of contractor's payment request percentages |

## PATTERNS TO FOLLOW

### Modal Pattern âœ…
```tsx
// Always include Cancel and Save/Submit buttons
<div className="flex justify-end gap-3">
  <button onClick={onClose} className="...">Cancel</button>
  <button onClick={handleSave} className="...">
    {loading ? 'Saving...' : 'Save'}
  </button>
</div>
```

### API Route Pattern âœ…
```tsx
// Always validate auth and handle errors
export async function POST(request: Request) {
  const { data: { session } } = await supabase.auth.getSession();
  if (!session) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  try {
    // ... logic
  } catch (error) {
    return NextResponse.json({ error: 'Failed' }, { status: 500 });
  }
}
```

### Don't Do This âŒ
```tsx
// Don't use direct Supabase calls from client without auth check
const { data } = await supabase.from('projects').select('*');

// Don't leave modals without Cancel buttons
// Don't hardcode colors - use CSS variables or theme utilities
```
